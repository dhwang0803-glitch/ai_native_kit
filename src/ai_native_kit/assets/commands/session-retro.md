현재 세션의 작업을 회고하고 하네스 개선점을 도출한다. 실행 계획 대비 실제 결과를 비교하고, 세션에서 잘 작동한 패턴과 실패한 패턴을 기록한다. 이 기록은 하네스 진화의 원료다.

> 하네스는 고정된 것이 아니라 매 세션마다 개선되어야 한다. 측정하지 않으면 평가할 수 없고, 평가하지 않으면 개선할 수 없다.

---

## Step 0. 세션 범위 확인

```bash
git branch --show-current
git log --oneline -20
```

이 세션에서 수행한 작업 범위를 파악한다:
- 어떤 브랜치에서 작업했는가?
- 몇 개의 커밋이 생성됐는가?
- 어떤 파일이 변경됐는가?

---

## Step 1. 실행 계획 대비 결과 비교

### 1-1. 실행 계획 찾기

```bash
# 최근 변경된 plan 파일
ls -lt docs/specs/plan/ 2>/dev/null
# 또는 세션 중 생성된 plan
git diff --name-only --diff-filter=A HEAD~10..HEAD | grep -i plan
```

실행 계획 파일이 없으면 → 사용자에게 "이 세션의 목표가 무엇이었는지" 확인 후 진행.

### 1-2. 계획 vs 실제 대조표

| Phase/단계 | 계획 | 실제 결과 | 일치 | 차이 원인 |
|-----------|------|----------|------|----------|

각 Phase에 대해:
- **완료**: 계획대로 구현됨
- **변경**: 구현했으나 계획과 다름 → 왜 다른가?
- **미완료**: 구현하지 못함 → 왜 못했는가?
- **계획 외**: 계획에 없었으나 추가로 구현함 → 왜 필요했는가?

### 1-3. 계획 품질 판정

| 판정 | 기준 |
|------|------|
| 계획이 좋았다 | 대부분 Phase가 계획대로 완료. 변경/미완료가 외부 요인(요구사항 변경 등) |
| 계획이 부족했다 | 변경/미완료가 계획 자체의 모호함이나 누락에서 기인 |
| 계획이 과했다 | Phase가 너무 세분화되어 불필요한 오버헤드 발생 |

---

## Step 2. 세션 패턴 분석

### 2-1. 잘 작동한 패턴 (Keep)

세션에서 효과적이었던 것들을 식별한다:
- 특정 프롬프트 패턴이 정확한 결과를 냈는가?
- 에이전트 순서(TDD 사이클 등)가 적절했는가?
- 컨텍스트 주입 시점이 적절했는가?
- 도구 사용이 효과적이었는가?

### 2-2. 실패한 패턴 (Drop)

세션에서 문제가 된 것들을 식별한다:
- 컨텍스트 과다로 모델이 혼란스러웠는가? (300K+ 토큰 시점)
- 자기 편향이 발생했는가? (자기 코드를 "잘 짰다"고 평가)
- 하드코딩/회피 패턴이 발생했는가?
- 아키텍처 드리프트가 누적됐는가?
- 도구 에러를 무시하고 진행했는가?

### 2-3. 시도할 패턴 (Try)

다음 세션에서 시도할 개선 사항:
- 컨텍스트를 더 일찍/늦게 주입해야 하는가?
- 에이전트 순서를 바꿔야 하는가?
- 새로운 도구/스킬이 필요한가?
- 교차 검증(`/cross-verify`)을 더 자주 사용해야 하는가?

---

## Step 3. 하네스 개선 제안

세션 분석 결과를 바탕으로 **구체적인 하네스 변경 제안**을 도출한다.

### 3-1. 변경 대상 분류

| 변경 대상 | 예시 |
|-----------|------|
| CLAUDE.md 규칙 | 새 금지 사항 추가, 컨벤션 수정 |
| 에이전트 템플릿 | 에이전트 프롬프트 수정, 새 에이전트 추가 |
| 슬래시 커맨드 | 커맨드 절차 수정, 새 커맨드 추가 |
| 도구/MCP | 새 도구 연결, 기존 도구 교체 |
| 문서 구조 | docs/ 폴더 구조, spec 템플릿 수정 |
| 훅 | pre-commit/post-checkout 수정, 세션 훅 추가 |

### 3-2. 제안 형식

각 제안에 대해:
```
### 제안 N: {제목}

- **유형**: {CLAUDE.md 규칙 / 에이전트 / 커맨드 / 도구 / 문서 / 훅}
- **근거**: 이 세션에서 {구체적 사건} 때문에 {구체적 문제}가 발생
- **변경 내용**: {무엇을 어떻게 바꿀지}
- **예상 효과**: {다음 세션에서 어떤 차이가 날지}
- **위험**: {이 변경의 부작용 가능성}
```

## Step 4. 회고 보고서 저장

### 보고서 형식

```markdown
# 세션 회고 — {날짜} {브랜치명}

## 세션 요약
- 브랜치: {branch}
- 커밋: {N}건
- 변경 파일: {N}개
- 교차 검증: 실행 여부 + 결과 요약

## 계획 vs 실제
{Step 1 대조표}

## 패턴 분석
### Keep (유지)
- {잘 된 것}

### Drop (중단)
- {문제된 것}

### Try (시도)
- {다음에 시도할 것}

## 하네스 개선 제안
{Step 3 제안들}

## 측정 지표
- 계획 이행률: {완료 Phase / 전체 Phase}%
- 교차 검증 불일치율: {불일치 / 전체 항목}%
- 자기 편향 발생 횟수: {N}회
- 아키텍처 드리프트 발생: 예/아니오
```

저장 위치: `docs/specs/plan/retro-{date}-{branch}.md`

---

## Step 5. 하네스 변경 적용 (사전 확인 필수)

1. 보고서를 **채팅에 먼저 제시**한다.
2. 사용자가 개선 제안 중 적용할 항목을 선택한다.
3. 선택된 항목만 실제 파일에 반영한다 (CLAUDE.md, 에이전트 템플릿 등).
4. 변경 사항은 커밋한다 (하네스 개선도 이력으로 남겨야 한다).

> ⚠️ 사용자 확인 없이 CLAUDE.md나 에이전트 템플릿을 수정하지 않는다.

---

## 활용 팁

- **매 세션 종료 시** `/session-retro`를 실행하면 하네스가 점진적으로 개선된다.
- 여러 세션의 회고를 모으면 **팀 차원의 하네스 개선** 근거가 된다.
- `/cross-verify` 결과가 있으면 회고 정확도가 높아진다 — 교차 검증 보고서를 참조한다.
- 계획 이행률이 계속 낮으면 → 계획 단계(spec-design)를 개선해야 한다.
- 자기 편향이 반복되면 → 교차 검증 빈도를 높여야 한다.
